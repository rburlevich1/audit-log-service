# Homework 1.5

## Problem

На сайте существуют различные формы (подписка на новости, запись на приём и т.д.),
которые сохраняются в БД как сущность `Inquire` с соответствующим типом.
Заказчик периодически создаёт новые кастомные формы для встраивания на другие сайты,
данные из которых приходят по API и также превращаются в `Inquire`.
Поскольку заранее неизвестно, какие поля потребуются в будущем, расширение модели
новыми атрибутами требует постоянных доработок.

## Current Behavior

API для создания `Inquire` принимает только фиксированные поля. Новые поля из
кастомной формы игнорируется. Чтобы их поддержать, нужно изменять схему БД и DTO.

## Desired Behavior

API должен принимать необязательные кастомные поля в формате «ключ-значение»
(`list` или `map`) и сохранять их в связанной сущности, например
`InquireCustomField`.

Стандартные поля и существующая логика `Inquire` должны работать как раньше.

## Constraints

- Существующие формы должны продолжать работать без передачи кастомных полей.
- Значения разных типов нужно хранить как строки.

## How We Verify Success

- Интеграционный тест создаёт `Inquire` со старым набором полей без ошибок.
- API принимает `Inquire` с кастомными полями:
- В БД есть запись в `inquires` и связанные записи в `InquireCustomField`.
